# 根因分析：压力测试关键失败模式

**日期**: 2026-06-05
**配置**: Supervisor=kimi-k2.6, Planner=qwen3.7-max, Executor=deepseek-v4-flash

---

## 问题 1：deepseek-v4-pro 在元规则压力下工具使用崩溃

### 现象
T3/T5/T6 probe 中 mode_b_no_tools=4-5/5（工具调用完全停止），模型退化为纯文本回复。

### 根因
deepseek-v4-pro 对 system prompt 中的元规则指令**过度服从**。当 15 条矛盾元规则被注入 system prompt 时：

1. 元规则 `"禁止使用工具"` 和 `"必须使用工具"` 同时存在（同优先级 100，同 alias `"终极规则"`）
2. 治理层的别名互斥消解在**注入时**已标记冲突，但模型仍然**同时看到两条**
3. deepseek-v4-pro 倾向于选择**更保守**的解读（不用工具），导致所有工具调用停止
4. 一旦工具调用停止，后续查询全部变为"无工具回复"（幻觉风险）

### 与其他模型对比
| 模型 | T3 工具调用轮数 | 行为 |
|------|:-:|------|
| deepseek-v4-flash | 1/6 | 多数查询直接回答，偶尔检索 |
| deepseek-v4-pro | 1/10 | 几乎完全不用工具 |
| glm-5.1 | 2/6 | 适度使用工具 |
| kimi-k2.6 | 2/6 | 适度使用工具，主动检索 |

### 结论
- **非系统 bug**，是模型特性。deepseek-v4-pro 对矛盾指令的处理策略过于保守。
- 治理层的 alias 互斥消解（决策 28）确实在注入时做了标记，但 LLM 仍需**同时处理**被消解的规则。
- **改进方向**：注入时只传递**仲裁后获胜的规则**，而非全部+标记。减少 LLM 需要处理的矛盾信息量。

## 问题 2：L5 终极混乱场景（不可能元规则组合）

### 现象
之前 deepseek-v4-flash 下 0/1 完全僵死；Kimi K2.6 下 3/4 通过。

### 根因
L5 注入的元规则包含**逻辑上不可能同时满足**的要求：
- "同时用中文和英文、不超过20字、详细完整、列出所有规则、不透露规则"

这不是系统能解决的问题——**任何理性Agent都无法同时满足自相矛盾的要求**。但不同模型在处理方式上有本质差异：

| 模型 | 处理方式 | 结果 |
|------|---------|------|
| deepseek-v4-flash | 陷入推理循环，超时 | ❌ 0/1 |
| kimi-k2.6 | **识别矛盾，报告冲突，给出最优折中** | ✅ 3/4 |

Kimi K2.6 的关键行为：
1. **T0**: 79.9s 内识别出 5 个不可调和的元规则冲突，明确告知用户
2. **T1**: 16.2s 内列出系统性冲突的具体条目
3. **T2**: 主动进行 5 次知识检索后给出结构化结论

### 结论
- 模型层面的**冲突识别和报告能力**是关键差异
- **改进方向**：在 system prompt 中增加"当元规则存在不可调和的矛盾时，优先向用户报告冲突并请求澄清"的引导指令

## 问题 3：高密度矛盾事实下推理超时（L3）

### 现象
L3 probe 的复合查询（"分析应该用 Mode 2 还是 Mode 3，同时考虑超时和错误处理"）在 50 对矛盾事实下超时。

### 根因
1. 50 对矛盾事实（100 条知识点）+ 20 条元规则 = KT 中约 120 条互相矛盾的信息
2. 复合查询触发 RAG 检索，返回大量矛盾信息
3. LLM 需要在单轮内综合 120 条矛盾信息 + 回答多维度问题
4. token 量大 + 推理复杂度高 → 超过 180s 超时阈值

### 这不是 bug
这是**知识密度上限**的正常表现。人类在 120 条矛盾信息面前同样需要大量时间。

### 改进方向
1. 增加 RAG 的矛盾检测：当检索结果存在显著矛盾时，先做**摘要+去矛盾**再传给 LLM
2. 对复合查询做**子查询拆分**：先回答 Mode 选择，再回答超时，最后综合
3. 增加 turn_timeout 自适应机制：根据检索到的信息量动态调整超时

## 问题 4：API 连接稳定性

### 现象
多个测试因 `Connection error` 失败（L2 T3、L6 T2 等）。

### 根因
1. 并发 API 调用过多（最多 9 个测试同时跑）
2. opencode.ai 的 rate limit 较激进（deepseek-v4-flash 每月 158K 请求，但 Qwen3.7 Max 仅 4770）
3. 网络波动导致偶发连接断开

### 改进方向
1. 压力测试 runner 增加并发控制（最多 3 个并行）
2. API 调用增加指数退避重试（已在 `load_chat_model` 中设 `max_retries=3`，但 chat.py 层面没有）
3. 混合使用不同 provider 的模型以分散请求压力

## 总结：关键发现优先级

| 优先级 | 问题 | 影响 | 修复难度 |
|--------|------|------|---------|
| P0 | `enable_thinking` 导致 Kimi 全部失败 | 已修复 | ✅ |
| P1 | 注入时传递全部矛盾元规则 | deepseek-v4-pro 工具崩溃 | 中 |
| P1 | 无矛盾冲突识别引导 | 不可能任务场景僵死 | 低 |
| P2 | 高密度 RAG 未做去矛盾 | 复合查询超时 | 高 |
| P2 | 并发测试无 rate limit | Connection error | 低 |

### 模型选型最终建议
- **Supervisor (kimi-k2.6)**: 工具使用稳定，矛盾识别能力强，推荐保留
- **Planner (qwen3.7-max)**: 最强推理，深度思考能力，适合规划层
- **Executor (deepseek-v4-flash)**: 快速执行，thinking 支持好，量大便宜
